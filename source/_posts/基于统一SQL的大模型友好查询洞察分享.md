### 1. Summary 概要

业界通过语义层直生成各库SQL和联邦引擎统一执行解决统一SQL查询场景，不依赖事后方言改写。本产品线当前"统一SQL+函数映射"方案存在等价函数依赖的结构性风险，需向语义层直接生成方言SQL演进。

### 2. Motivation 动机

借鉴业界跨库查询统一方案，明确本产品线SQL拉齐的技术演进方向。

1、产品现状：五个子域跨StarRocks/Druid/GaussDB三库查询，AI需生成一种SQL让三库都能执行

当前方言改写流程：

```
用户自然语言: "查询最近24小时的数据"
        ↓
    BI报表的语义层（非标准语义层，会通过编译器翻译成guass sql）
        ↓
AI基于表模型定义生成统一SQL:
  SELECT * FROM metrics
  WHERE ts_long > UNIX_TIMESTAMP(CURRENT_TIMESTAMP - INTERVAL 1 DAY) * 1000
        ↓
引擎层方言改写，分别下推各库执行:
  ┌─ StarRocks:  零改写(原生支持UNIX_TIMESTAMP)
  │              → 执行成功 ✅ 返回id=3,4,5
  │
  ├─ GaussDB:    零改写(原生支持UNIX_TIMESTAMP)
  │              → 执行成功 ✅ 返回id=3,4,5
  │
  ├─ Druid报文:  FROM_UNIXTIME函数不下推 → 内存计算报错
  │              → 执行失败 ❌ 场景失败
  │
  └─ Druid SQL:  函数映射改写为:
                  WHERE ts_long > CAST(TIMESTAMP_TO_MILLIS(CURRENT_TIMESTAMP - INTERVAL 1 DAY) / 1000 AS INTEGER) * 1000
                  → 执行成功 ✅(降级至Druid SQL路径)
```

    a、当前方案：AI生成统一SQL + 引擎层函数映射改写 + 下推各库执行
    b、已知问题：Druid报文层不支持FROM_UNIXTIME等函数下推，导致AI问数存在部分场景查询报错
    c、结构性风险：方案依赖每个数据库均存在等价函数可映射，新增不支持等价函数的库即失效

2、业界已有多条成熟路线解决跨库SQL适配，需对比研判正确演进方向

3、本产品线当前方案已验证存在结构性风险（Druid报文场景失败），需尽快明确演进方向

### 3. 洞察内容

> 主要分析业界跨库SQL适配的三种路线、各自如何避免等价函数依赖，以及本产品线应如何演进

### 3.1 业界三种路线 vs 本方案：SQL怎么到各库执行

**本方案（方言翻译）——当前现状**

```
用户: "查询最近24小时数据"
        ↓
AI生成统一SQL（不感知目标库）
        ↓
引擎层函数映射改写，分别下推各库:
  ┌─ StarRocks:  不改(原生支持)         → ✅ 执行成功
  ├─ GaussDB:    不改(原生支持)         → ✅ 执行成功
  ├─ Druid报文:  函数不下推             → ❌ 执行失败
  └─ Druid SQL:  改写为等价函数         → ✅ 执行成功(降级路径)
```

问题：生成统一SQL后再改写，某库不支持等价函数即失败。

**路线一：联邦引擎——不推SQL给源库，引擎层统一执行**

```
用户: "查询最近24小时数据"
        ↓
Trino收到查询请求
        ↓
不推SQL给源库，拉取数据到引擎层:
  ┌─ 从StarRocks拉取数据 ──→ Trino引擎层
  ├─ 从Druid拉取数据     ──→ Trino引擎层
  └─ 从GaussDB拉取数据   ──→ Trino引擎层
        ↓
Trino引擎统一执行查询、JOIN、聚合
        ↓
返回结果
```

* 为什么不依赖等价函数：SQL不在源库执行，源库只需吐数据，不需要支持任何特定函数
* 代表：Meta日处理数千万查询、EB级数据 [1]；Uber 15集群、5000+节点、日50万查询
  **路线二：语义层——定义指标一次，感知目标库自动生成该库SQL**

```
用户/AI: "查CPU均值"
        ↓
语义层（指标定义：CPU均值 = 按月聚合CPU平均值）
        ↓
感知目标库，自动生成各库专属SQL:
  ┌─ StarRocks → 生成StarRocks方言SQL → 执行 ✅
  ├─ Druid     → 生成Druid方言SQL     → 执行 ✅
  └─ GaussDB   → 生成GaussDB方言SQL   → 执行 ✅
        ↓
返回结果（生成时就对了，无需改写）
```

* 为什么不依赖等价函数：每个库拿到的是语义层为自己专属生成的SQL，不存在"映射不了"的问题
* 代表：dbt MetricFlow、Cube、Netflix DataJunction、Airbnb Minerva
* 对大模型的友好性：AI只需声明查什么指标，语义层自动完成方言适配，降低AI生成SQL的复杂度
* 局限：指标建模投入较大（中型企业数百指标=数月建模）

**路线三：引擎统一——只用一个引擎，消除方言问题**

```
用户: "查CPU均值"
        ↓
AI生成SQL（只有一种方言，因为只有一个引擎）
        ↓
统一引擎(如StarRocks)直接执行
        ↓
返回结果（不存在跨库问题）
```

* 为什么不依赖等价函数：只有一个引擎、一种方言，不存在跨库映射
* 代表：Meta四引擎→Presto一个 [2]，节省"数十万行代码和数千台机器"；Velox集成Presto实现3-10x效率提升 [2]。滴滴四引擎→StarRocks，查询性能4x、总成本降80%、集群60+→10节点以内
* 教训：性能问题可以用工程手段解决，但运维复杂度无法用工程手段消除——这是引擎统一的根本驱动力 [2]
* 局限：迁移成本高，已有异构库存量场景不适用

### 3.2 三种路线对比与选型

| 维度        | 联邦引擎(Trino)        | 语义层(dbt/Cube)          | 引擎统一(Meta/滴滴) | 本方案(方言翻译)       |
| --------- | ------------------ | ---------------------- | ------------- | --------------- |
| 适配方式      | 不推SQL给源库           | 感知目标库直接生成              | 单引擎无跨库        | 统一SQL+运行时改写     |
| 依赖源库SQL能力 | 否                  | 否                      | 否             | **是**           |
| 等价函数依赖风险  | 无                  | 无(生成时就对了)              | 无(单引擎)        | **有(Druid已失败)** |
| 跨源JOIN    | 支持                 | 不支持                    | 不支持           | 不支持             |
| SQL下推执行   | 否(引擎层执行)           | 是                      | N/A           | 是               |
| 迁移成本      | 中                  | 中                      | 高             | 低               |
| 适用本产品线    | ❌ 单数据源查询更慢，需额外部署集群 | ✅ 三库不动，成本最低，可并行过渡      | ❌ 三库功能互补不可合并  | 当前方案(有风险)       |
| 代表实践      | Meta/Uber          | Netflix/Airbnb/Spotify | Meta/滴滴       | 本产品线            |

**选型结论：业界三种路线均不依赖事后方言改写，本产品线选型如下：**

1、联邦引擎——不采用。本产品线查询以单数据源为主，不需要跨源JOIN；联邦引擎拉取数据到引擎层执行，单数据源查询比原生更慢；需额外部署维护Trino集群

2、引擎统一——不采用。三库功能互补（StarRocks高性能OLAP、Druid时序实时、GaussDB关系型配置），单一引擎无法同时最优；迁移三个生产库业务中断风险大

3、语义层——采用。三库不动迁移成本最低；感知目标库直接生成该库SQL，消除等价函数依赖；可与当前方言翻译并行运行，平滑过渡

### 3.3 方言翻译的结构性风险

本产品线当前方案分三步，三个风险均发生在第二步：

```
Step 1: AI生成统一SQL（不感知目标库）
        ↓
Step 2: 引擎层函数映射改写          ← ⚠️ 三个风险均在此步骤
        改写规则: 为每个库查找等价函数并替换
        ┌─ StarRocks: 原生支持该函数       → 改写成功 ✅
        ├─ GaussDB:   原生支持该函数       → 改写成功 ✅
        ├─ Druid报文: 不支持该函数下推     → 改写失败 ❌ (报文场景)
        └─ Druid SQL: 找到等价函数可映射   → 改写成功 ✅ (降级路径)
        ↓
Step 3: 改写后的SQL下推各库执行

⚠️ 风险①  等价函数依赖:  每个库必须存在等价函数可替换，否则Step 2失败
          → 已发生: Druid报文无等价函数下推，场景失败

⚠️ 风险②  语义差异不可处理: 仅能做函数名替换，无法处理NULL聚合行为、
          时区默认值、时间精度等语义级差异

⚠️ 风险③  N×M维护负担: 每个新函数×每个新数据库需新增一条映射规则
          → 5个函数×4个数据源=20条规则，新增1个库则变为5×5=25条
```

当前方言改写器基于Apache Calcite框架实现函数映射。Druid报文层场景失败已验证风险①，风险②③尚未触发但属结构性隐患。

### 4. 我们现状与下一步

**As-Is → To-Be 架构变化：**

```
【As-Is 当前架构】

用户/AI: "查最近24小时数据"
  ↓
AI生成统一SQL（不感知底层是哪个数据库）
  ↓
引擎层函数映射改写  ← ⚠️ 风险在此
  ├─ StarRocks:  查找等价函数 → 找到 → 改写 → 执行 ✅
  ├─ GaussDB:    查找等价函数 → 找到 → 改写 → 执行 ✅
  ├─ Druid报文:  查找等价函数 → 找不到 → ❌ 失败
  └─ Druid SQL:  查找等价函数 → 找到 → 改写 → 执行 ✅(降级)

问题: AI不感知底层用哪个数据库，生成统一SQL后再逐库尝试改写
     改写时依赖每个库都有等价函数，找不到就失败


【To-Be 目标架构】

用户/AI: "查CPU均值"
  ↓
语义层（指标定义: CPU均值 = 按月聚合CPU平均值）
  ↓
根据指标定义追溯到底层表 → 查路由映射确定目标数据库
  ┌─ 路由映射: metrics表 → StarRocks → 生成StarRocks方言SQL → 执行 ✅
  ├─ 路由映射: alerts表  → Druid     → 生成Druid方言SQL     → 执行 ✅
  └─ 路由映射: config表  → GaussDB   → 生成GaussDB方言SQL   → 执行 ✅

优势: 语义层通过路由映射感知底层数据库，生成时就对了，不存在"改不动"
```

**核心变化：从"不感知底层→统一生成→事后改写"到"感知底层→直接生成该库SQL"。**

感知机制：语义层维护表-数据库路由映射（metrics表→StarRocks、alerts表→Druid）。AI声明指标后，语义层追溯到底层表，查路由映射确定目标库，直接生成该库SQL。当前BI的SQL编译器写器无此能力，只能统一生成后让查询引擎方言改写。

业界验证：Netflix、Airbnb、Spotify均已落地。OSI Initiative 40+公司参与，语义层正成为产业标准 [3]。

### 5. Conclusion 观点

1、当前"统一SQL+方言改写"方案依赖各库存在等价函数，Druid报文场景失败已验证此风险。业界三种路线均不依赖事后方言改写 [1][2][3]。

2、大模型查询统一SQL向语义层感知底层数据库映射数据库SQL，从根源消除"改不动"的风险。
